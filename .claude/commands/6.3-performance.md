## Performance-audit

**Scope**: $ARGUMENTS (default: hele `src/`)

Kjør `nextjs-performance-auditor`-agenten mot gitt scope. Brukes når brukeren opplever treghet, før deploy til prod, eller periodisk (kvartalsvis).

### Workflow

1. Bekreft scope.
2. Kjør `pnpm build` først for å få ferske bundle-tall (hvis ikke allerede gjort). Logg output.
3. Kall `Agent({subagent_type: "nextjs-performance-auditor", prompt: "Audit <scope>. Bruk build-output: <paste>. Følg sjekklisten i nextjs-performance-auditor.md."})`.
4. Lagre rapport til `teknisk/dokumentasjon/audits/YYYY-MM-DD-performance.md`.

### Etter static audit

Claude kan ikke måle runtime. Anbefal følgende for live-målinger:

- **Lighthouse** i Chrome DevTools på dev-server (og prod-URL hvis tilgjengelig)
- **Vercel Analytics** for Core Web Vitals fra ekte brukere
- **`@next/bundle-analyzer`**: `ANALYZE=true pnpm build` for visualisering av bundle
- **`next-devtools`-MCP**: real-time runtime-info fra dev-server

### Tolkning av funn

- **> 300 KB First Load JS per rute**: flag for refactor. Bruk `next/dynamic` eller flytt til Server Component.
- **LCP > 2.5s**: sjekk `priority` på hero-image, font-loading, tunge client-komponenter above the fold.
- **CLS > 0.1**: bilder mangler `width`/`height`, font-swap uten `swap`-strategi, dynamisk innsatt innhold uten reservert space.
- **TBT > 200ms**: for mye client-JS eller hydration-cost. Flytt logikk til server.
