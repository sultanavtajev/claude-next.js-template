## Prosjekt-discovery

**Startkontekst**: $ARGUMENTS (brukes som utgangspunkt, ellers spør Claude først)

Denne kommandoen leder en grundig discovery-samtale og produserer 10 strukturerte markdown-dokumenter som Claude senere bruker som prosjekt-kontekst i alle oppgaver.

## Viktig — arbeidsregler

1. **All interaksjon via `AskUserQuestion`-verktøyet** — ikke fritekst-spørsmål. Gruppér spørsmål logisk (maks 4 per kall, 2–4 valg per spørsmål).
2. **Bruker kan hoppe over** — hvis noe er uklart, fyll feltet med `TBD` og flagg i `09-aapne-sporsmal.md`.
3. **Iterér, ikke perfeksjoner** — første versjon er et utgangspunkt. Bruker kan kjøre `/7.0-discovery` på nytt senere for å oppdatere.
4. **Bruk eksisterende kontekst** — `CLAUDE.md` har prosjektnavn, stack, locales, design system. Ikke spør om ting som allerede er fastlagt.

## Mappe- og filstruktur

Opprett `teknisk/dokumentasjon/prosjekt/` (hvis ikke finnes) og produser:

```
teknisk/dokumentasjon/prosjekt/
├── 00-visjon.md
├── 01-malgruppe.md
├── 02-scope.md
├── 03-funksjonelle-krav.md
├── 04-ikke-funksjonelle-krav.md
├── 05-user-flows.md
├── 06-datamodell.md
├── 07-arkitektur.md
├── 08-roadmap.md
└── 09-aapne-sporsmal.md
```

Hver fil har en fast struktur (se nedenfor). Claude fyller inn basert på AskUserQuestion-svar.

## Workflow

Gå gjennom fase-for-fase. Rapporter underveis ("Fase 2/10 ferdig..."). La brukeren pause eller stoppe når som helst.

---

### Fase 00 — Visjon

**Fil**: `00-visjon.md`

Spørsmål (ett AskUserQuestion-kall med 4 spørsmål):

1. **Hva er kort beskrivelse av prosjektet?** (Other-fritekst, brukes som "elevator pitch")
2. **Hvilket konkret problem løser dette?** (Other-fritekst)
3. **Hvorfor er det verdt å bygge?** Valg: "Reduserer manuell jobb", "Øker inntekt", "Forbedrer brukeropplevelse for eksisterende problem", "Tester en hypotese om markedet"
4. **Hva er suksesskriteriet?** Valg: "Brukerdelving (antall aktive)", "Inntekt (ARR/MRR)", "Kvalitativ tilbakemelding", "Intern effektivitet", "Teknologisk demonstrasjon"

Output-mal:
```markdown
# 00 — Visjon

## Prosjekt-beskrivelse (én setning)
<svar>

## Problem vi løser
<svar>

## Hvorfor dette er verdt å bygge
<svar>

## Suksesskriterium
<svar>

## Elevator pitch (30 sek)
<generér basert på de fire svarene — kort og konkret>
```

### Fase 01 — Målgruppe

**Fil**: `01-malgruppe.md`

Spørsmål (ett AskUserQuestion med 3 spørsmål):

1. **Hvem er primær-bruker?** Valg: "B2B — ansatte i bedrifter", "B2C — sluttbrukere", "Utviklere", "Interne ansatte i min organisasjon"
2. **Hva er deres tekniske nivå?** Valg: "Lavt (trenger enkel UI)", "Middels", "Høyt (utviklere/power-users)"
3. **Hva er deres viktigste behov akkurat nå?** (Other-fritekst)

Oppfølging hvis B2B eller B2C: ett AskUserQuestion til med aldersgruppe, geografisk marked, og kjent alternativ de kanskje bruker i dag.

Output-mal:
```markdown
# 01 — Målgruppe

## Primær-persona
- **Rolle**: <svar>
- **Teknisk nivå**: <svar>
- **Geografi**: <svar>
- **Viktigste behov**: <svar>
- **Hva de bruker i dag**: <svar>

## Sekundær-persona (hvis relevant)
<fyll ut eller "ingen — vi fokuserer kun på primær">

## Anti-persona
<hvem dette IKKE er for — skjerper fokus>
```

### Fase 02 — Scope

**Fil**: `02-scope.md`

Spørsmål (ett AskUserQuestion med 3 spørsmål, oppfølging via fritekst):

1. **Hva er "absolutt minimum" for MVP?** (Other-fritekst, 2–4 kjernefeatures)
2. **Hva kan vente til v2?** (Other-fritekst)
3. **Hva vil vi EKSPLISITT ikke bygge?** (Other-fritekst — ikke-mål er like viktige)

Output-mal:
```markdown
# 02 — Scope

## MVP (v1) — må være med før launch
- <feature 1>
- <feature 2>
- ...

## Post-MVP (v2+) — planlagt men ikke nå
- <feature>
- ...

## Ikke-mål — dette bygger vi IKKE
- <ting>
- ...

## Begrunnelse for scope-kuttene
<kort forklaring — hvorfor vi sier nei>
```

### Fase 03 — Funksjonelle krav

**Fil**: `03-funksjonelle-krav.md`

Dette er det mest iterative steget. For hver MVP-feature fra fase 02, still oppfølgingsspørsmål:

For hver feature, ett AskUserQuestion med 3 spørsmål:
1. **Hva skal brukeren kunne gjøre?** (fritekst — user story-format: "Som X vil jeg Y slik at Z")
2. **Hvordan vet vi at det funker?** (akseptkriterier — fritekst, 2–3 punkter)
3. **Prioritet?** Valg: "Must-have", "Should-have", "Could-have"

Fortsetter til alle MVP-features er dekket. Rapporter "Feature 3/7 ferdig..." underveis.

Output-mal:
```markdown
# 03 — Funksjonelle krav

## Feature: <navn>
**Prioritet**: Must / Should / Could

**User story**: Som <rolle> vil jeg <handling> slik at <verdi>.

**Akseptkriterier**:
- [ ] <kriterium 1>
- [ ] <kriterium 2>

**Avhengigheter**: <andre features som må være på plass først, eller "ingen">

---

## Feature: <neste>
...
```

### Fase 04 — Ikke-funksjonelle krav

**Fil**: `04-ikke-funksjonelle-krav.md`

Ett AskUserQuestion med 4 spørsmål:

1. **Forventet brukervolum ved MVP-launch?** Valg: "<100 brukere (intern/test)", "100-1k", "1k-10k", "10k+ (skala kritisk fra dag 1)"
2. **Pålitelighetskrav?** Valg: "Best-effort (OK med korte avbrudd)", "Høy tilgjengelighet ønskelig (99.5%)", "Kritisk infrastruktur (99.9%+)"
3. **Sikkerhetsnivå?** Valg: "Lav (ingen sensitive data)", "Middels (brukerkontoer, normal GDPR)", "Høy (finansielle/helse-data)", "Enterprise (SOC2/ISO27001 kreves)"
4. **Språk-strategi?** Valg: "Bare default locale", "Default + engelsk", "Nordisk", "Global (5+ språk)"

Output-mal:
```markdown
# 04 — Ikke-funksjonelle krav

## Ytelse
- **Forventet volum**: <svar>
- **Response time-mål**: <typisk <200ms p95>
- **Core Web Vitals-mål**: LCP < 2.5s, CLS < 0.1

## Skala
- **Samtidige brukere**: <estimat>
- **Datavolum**: <estimat>

## Pålitelighet
- **Uptime-mål**: <svar>
- **RPO/RTO** (hvis kritisk): <fyll ut>

## Sikkerhet
- **Nivå**: <svar>
- **Compliance**: <GDPR, HIPAA, osv.>
- **Spesielle krav**: <f.eks. SSO, 2FA, audit log>

## Tilgjengelighet (a11y)
- **WCAG-nivå**: AA som minimum
- **Tastaturnavigasjon**: full støtte

## Internasjonalisering
- **Locales**: <fra i18n-config i CLAUDE.md>
- **Høyre-mot-venstre**: <ja/nei>
```

### Fase 05 — User flows

**Fil**: `05-user-flows.md`

For hver MVP-feature, still ett AskUserQuestion med 2 spørsmål:

1. **Hva er typisk steg-for-steg for brukeren?** (fritekst)
2. **Hva er viktigste edge cases?** Valg: "Feil/timeout ved nettverkskall", "Tom tilstand (ingen data)", "Uautorisert bruker", "Konkurrerende endringer (noen andre redigerer samtidig)"

Output-mal:
```markdown
# 05 — User flows

## Flow: <navn>

### Happy path
1. Bruker <handling>
2. System <respons>
3. ...

### Edge cases
- **<scenarie>**: <hvordan håndteres>

### Viktige beslutninger
- <f.eks. "vi viser toast ved suksess, redirect først etter 1 sek">

---

## Flow: <neste>
...
```

### Fase 06 — Datamodell

**Fil**: `06-datamodell.md`

Ett AskUserQuestion om hoved-entiteter, deretter oppfølging per entitet:

1. **Hvilke er hovedentitetene?** (fritekst, komma-separert, f.eks. "users, posts, comments, tags")

For hver entitet, ett AskUserQuestion med 2 spørsmål:
1. **Viktigste felter?** (fritekst)
2. **Relasjoner til andre entiteter?** (fritekst)

Output-mal:
```markdown
# 06 — Datamodell

## Entiteter

### users
- `id`: uuid (PK, ref auth.users)
- `email`: text (unique, not null)
- `created_at`: timestamptz
- ...

Relasjoner:
- 1:N posts (via `posts.author_id`)

---

### posts
- `id`: uuid (PK)
- `author_id`: uuid (FK → users.id, cascade)
- `title`: text (not null)
- `body`: text (not null)
- `created_at`: timestamptz
- `updated_at`: timestamptz

Relasjoner:
- N:1 users
- 1:N comments
- N:N tags (via post_tags)

---

## RLS-strategi

- `users`: les egen + offentlig profil-felt
- `posts`: eier-RLS (`author_id = auth.uid()`)
- ...

## Indekser vi trenger

- `posts(author_id, created_at DESC)` for brukerens posts-side
- ...
```

Dette er en **skeleton** for kommende `supabase-migrations` — skal videreforedles når første migrasjon skrives.

### Fase 07 — Arkitektur

**Fil**: `07-arkitektur.md`

Ett AskUserQuestion med 3 spørsmål:

1. **Hvilke eksterne integrasjoner trenger MVP?** Valg: "Stripe/Vipps betaling", "Resend/Postmark e-post", "Sentry/Datadog monitoring", "Ingen — selvstendig MVP"
2. **Host-strategi?** Valg: "Vercel + Supabase (default)", "Self-hosted på Digital Ocean/AWS", "Hybrid (Vercel frontend + egen backend)"
3. **Background jobs?** Valg: "Ingen trengs", "Supabase pg_cron + edge functions", "Ekstern (Inngest, Trigger.dev)", "Må undersøkes"

Output-mal:
```markdown
# 07 — Arkitektur

## Stack (fra CLAUDE.md)
Referer til CLAUDE.md "Stack"-seksjonen.

## Integrasjoner (MVP)
- <tjeneste>: <formål, f.eks. "Stripe for abonnement-betaling">
- ...

## Integrasjoner (post-MVP)
- ...

## Background jobs
<strategi — f.eks. "pg_cron for daglige digest-emails">

## Arkitektur-beslutninger (ADR-log)

Lagres i `teknisk/dokumentasjon/beslutninger/` etter hvert som de tas.

Første beslutning:
- **ADR-001**: <tittel, f.eks. "Valg av Supabase framfor Postgres-self-hosted"> — <kort begrunnelse>
```

1. **Tidshorisont for MVP?** Valg: "1 uke", "2-4 uker", "1-3 måneder", "Åpent"

```markdown
# 08 — Roadmap

**Mål**: <kort>
**Features**:
- <Should-have fra scope>

## Fase 3+ — Fremtid
<løst definert>
```

### Fase 09 — Åpne spørsmål

**Fil**: `09-aapne-sporsmal.md`

Claude går gjennom det som er markert TBD i tidligere filer + spør eksplisitt:

1. **Hva er de tre største tekniske risikoene?** (fritekst)
2. **Hvilke antagelser må vi validere tidlig?** (fritekst)
3. **Hvem trenger vi avklaring fra?** (fritekst — stakeholders, juridisk, design, osv.)

Output-mal:
```markdown
# 09 — Åpne spørsmål og risikoer

## Tekniske risikoer
1. <risiko> — mitigering: <plan>
2. ...

## Antagelser som må valideres
- <antagelse> — hvordan vi validerer: <test/research-plan>

## Avklaringer trengs fra
- <stakeholder>: <hva vi trenger å vite>

## TBD-felter fra andre dokumenter
- `01-malgruppe.md`: sekundær-persona
- `04-ikke-funksjonelle-krav.md`: compliance-detaljer
- ...
```

## Når ferdig

Rapportér til bruker:
```
✓ Discovery ferdig — 10 filer opprettet i teknisk/dokumentasjon/prosjekt/

Anbefalt neste steg:
1. Les gjennom filene, juster TBD-felter
2. Begynn på første feature via /2.0-task <første must-have>
3. Kjør /7.0-discovery på nytt når scope endres eller nye antagelser valideres
```

Legg også til en note i `CLAUDE.md` (eller bekreft at den er der): Claude bør lese `teknisk/dokumentasjon/prosjekt/00-visjon.md` og `03-funksjonelle-krav.md` ved store nye oppgaver for kontekstbevissthet.

## Re-kjøring

Bruker kan kjøre `/7.0-discovery` flere ganger:
- Ved scope-endringer: oppdater `02-scope.md` og `03-funksjonelle-krav.md`
- Ved nye personas: legg til i `01-malgruppe.md`
- Ved milestones: oppdater `08-roadmap.md`

Ved re-kjøring: spør bruker om å identifisere **hva som endrer seg**, ikke kjør alle 10 faser på nytt.
