## Prosjekt-discovery (research-drevet)

**Startkontekst**: $ARGUMENTS

Denne kommandoen tar brukeren gjennom 10 planleggingsfaser. For **hver** fase:

1. **Claude spawner en research-subagent** som får kontekst fra tidligere faser, gjør web-research om bransjen / domenet / beste praksis
2. **Agenten returnerer 3 anbefalte alternativer** med kort begrunnelse for hver
3. **Claude presenterer anbefalingene via `AskUserQuestion`** — brukeren velger ett eller skriver eget via "Other"
4. **Svaret lagres**, konteksten vokser, neste fase starter rikere enn forrige

## Arbeidsregler

1. **All interaksjon via `AskUserQuestion`** — aldri fritekst.
2. **Alltid spawn research-agent først** — ikke improviser alternativer uten evidens.
3. **Agent-spawn pattern**: `Agent({subagent_type: "general-purpose", prompt: "<fase-spesifikk prompt med all prior kontekst>"})`
4. **Bruker kan hoppe over** — velg "Ingen passer, vi utsetter" → felt blir TBD, flagg i `09-aapne-sporsmal.md`.
5. **Gjenbruk tidligere svar** — spar bruker for duplisering. Ikke spør om ting CLAUDE.md allerede har (prosjektnavn, stack, locales).

## Mappe- og filstruktur

```
teknisk/dokumentasjon/prosjekt/
├── 00-visjon.md                # beslutning + valgt begrunnelse
├── 01-malgruppe.md
├── 02-scope.md
├── 03-funksjonelle-krav.md
├── 04-ikke-funksjonelle-krav.md
├── 05-user-flows.md
├── 06-datamodell.md
├── 07-arkitektur.md
├── 08-roadmap.md
├── 09-aapne-sporsmal.md
└── research/                   # full agent-research per fase
    ├── 00-visjon.md            # alle 3 alternativer, kilder, full analyse
    ├── 01-malgruppe.md
    ├── 02-scope.md
    ├── 03-funksjonelle-krav.md
    ├── 04-ikke-funksjonelle-krav.md
    ├── 05-user-flows.md
    ├── 06-datamodell.md
    ├── 07-arkitektur.md
    └── 08-roadmap.md
```

**Hvorfor to filer per fase**:
- **Hoved-fil** (`prosjekt/<fase>.md`): beslutningen som ble tatt + begrunnelse for den. Kort, kuratert.
- **Research-fil** (`prosjekt/research/<fase>.md`): full output fra research-agenten — alle 3 alternativer, web-kilder, konkurrenter undersøkt, alternativer som ble forkastet. Brukes for oppfølging, re-vurdering, eller hvis beslutningen skal revurderes senere.

Fase 09 (åpne spørsmål) har ikke separat research-fil siden den ER en syntetisering på tvers av alle tidligere.

## Per-fase workflow-mal

For hver fase, følg nøyaktig denne sekvensen:

```
1. Oppsummér kontekst fra tidligere faser (hva vi vet nå)
2. Spawn research-agent (Agent-verktøy) — be om strukturert output:
   a. 3 anbefalte alternativer
   b. For hver: label, description, full reasoning, trade-offs
   c. Liste over undersøkte konkurrenter / referanser / web-kilder
   d. Alternativer som ble vurdert og forkastet, med begrunnelse
3. LAGRE agent-rapporten til prosjekt/research/<fase>.md (full output, ikke kuratert)
4. AskUserQuestion med de 3 anbefalingene som options + "Other"
5. Bruker velger (eller "Other" for eget svar)
6. Skriv hoved-fil prosjekt/<fase>.md med:
   - Valgt alternativ + begrunnelse (kort)
   - Lenke til research-filen: "Full research: [research/<fase>.md](./research/<fase>.md)"
7. Rapport: "Fase NN/09 ferdig ✓ (research lagret til prosjekt/research/<fase>.md)"
```

Hver `prosjekt/research/<fase>.md` skal ha struktur:

```markdown

- <URL 1> — <hva vi fant>
- <URL 2> — <hva vi fant>

## Konkurrenter / referanser undersøkt

```

### Fase 00 — Visjon

**Kontekst fra CLAUDE.md**: prosjektnavn + kort beskrivelse er allerede satt. Bygg videre på det.

**Research-agent prompt**:
```
Du er en produkt-strategist. Basert på prosjekt-beskrivelsen: "<PROJECT_DESCRIPTION fra CLAUDE.md>"
og brukerens innspill: "<$ARGUMENTS hvis gitt>".

1. Søk etter 2-3 lignende produkter/konkurrenter i samme nisje (bruk WebSearch).
2. Analysér hva deres kjerne-verdiforslag er.
3. Foreslå 3 ulike visjons-vinklinger for vårt prosjekt. For hver:
   - En kort én-setnings pitch (label)
   - Begrunnelse (3-5 setninger): hvorfor denne vinkelen, hvilken konkurrent den differensierer mot
   - Trade-offs
   - Hvilken suksess-metrikk som passer

4. List minst 2 alternativer du vurderte men forkastet, med begrunnelse.

Returnér rapport i følgende markdown-struktur (matcher research-fil-malen i kommandoen):
- ## Kontekst sendt til agent
- ## Web-research utført (med URLer)
- ## Konkurrenter / referanser undersøkt
- ## Anbefalte alternativer (1, 2, 3 med full struktur per research-fil-mal)
- ## Forkastede alternativer
- ## Åpne spørsmål

Total lengde: maks 800 ord.
```

Claude lagrer hele agent-outputen til `teknisk/dokumentasjon/prosjekt/research/00-visjon.md` før AskUserQuestion kalles.

**AskUserQuestion**:
```
Question: "Hvilken visjons-vinkling passer best?"
Header: "Visjon"
Options (fra agentens 3 forslag — bruk agentens reasoning som description):
- <Alt 1 label>: <agentens reasoning>
- <Alt 2 label>: <agentens reasoning>
- <Alt 3 label>: <agentens reasoning>
- Other: "Jeg har en annen vinkel — jeg skriver selv"
```

**Fil-output** (`00-visjon.md`):
```markdown
# 00 — Visjon

## Valgt vinkling
<label fra valgt alternativ>

## Pitch (én setning)
<fra agent eller bruker>

## Begrunnelse
<agentens reasoning eller brukerens fritekst>

## Differensierende mot
- <konkurrent 1> — ulikhet: <kort>
- <konkurrent 2> — ulikhet: <kort>

## Suksess-metrikk
<valgt metrikk>

## Elevator pitch (30 sek)
<generér basert på alt over — kort og konkret>

---

**Full research**: [research/00-visjon.md](./research/00-visjon.md) — inkluderer alle alternativer (også forkastede), kilder, konkurrenter undersøkt.
```

### Fase 01 — Målgruppe

**Research-agent prompt**:
```
Basert på visjonen fra fase 00: "<innhold fra 00-visjon.md>"

Research hvem denne type produkt typisk brukes av:
1. Søk på nettet etter demografi / personas for lignende produkter.
2. Vurder 2-3 kandidat-personas som ville hatt mest nytte.
3. For hver persona: rolle, kontekst (jobb/privat), tekniske nivå, smertepunkter, verdi de får.

Foreslå 3 ulike primær-persona-valg, med reasoning for hvorfor hver ville vært god førsteprioritet for MVP.
```

**AskUserQuestion**: velg primær-persona fra 3 anbefalte + "Other".

**Oppfølging**: ett AskUserQuestion til med 3 spørsmål basert på valgt persona:
- Antatt volum av denne gruppen?
- Hvilke alternativer bruker de i dag?
- Hvor finner vi dem (markedskanaler)?

**Fil-output** (`01-malgruppe.md`): struktur som i fase-00-malen, med personas.

### Fase 02 — Scope (MVP)

**Research-agent prompt**:
```
Gitt visjon: "<00>" og primær-persona: "<01>".

1. Søk etter "MVP patterns" for denne type produkt.
2. Identifisér minste levedyktige sett av features som faktisk løser kjerneproblemet.
3. Foreslå 3 scope-strategier:
   - "Tynn vertikal" (bare core-use-case, dypt gjennomført)
   - "Horisontal basis" (flere features, hver overfladisk)
   - "Nisje-versjon" (løs ett konkret del-problem komplett)

For hver: begrunnelse (2-3 setninger), estimert tidsbruk til MVP, risiko ved valget.
```

**AskUserQuestion**: velg scope-strategi.

**Oppfølging**: iterér gjennom features. For hver:
- Agent foreslår 3-5 kandidat-features basert på valgt strategi
- Bruker velger hvilke som er Must / Should / Could / Won't (via AskUserQuestion multiSelect=true per kategori)

**Fil-output** (`02-scope.md`): liste med Must/Should/Could/Won't + begrunnelse.

---

### Fase 03 — Funksjonelle krav

**Research-agent prompt per feature** (fra 02-scope Must-have):
```
Feature: "<feature-navn>"
Prosjekt-kontekst: <visjon + persona>

1. Søk etter "best practices" for denne type feature.
2. Identifisér 3 vanlige måter å implementere det på (simplest, medium, rich).
3. For hver: user story, akseptkriterier (3-5 punkter), antatt kompleksitet.
```

**AskUserQuestion per feature** (multi-batch):
- Hvilken implementasjons-nivå?
- Hvilke akseptkriterier er must-have?

**Fil-output** (`03-funksjonelle-krav.md`): strukturert liste per feature.

---

### Fase 04 — Ikke-funksjonelle krav

**Research-agent prompt**:
```
Gitt prosjekt-kontekst (visjon + persona + scope):

1. Hvilke ikke-funksjonelle krav er kritiske for denne type app? (Ytelse, skala, sikkerhet, compliance.)
2. Hva er bransje-standard for lignende produkter?
3. For hver dimensjon (ytelse, skala, sikkerhet, tilgjengelighet, i18n), foreslå 2-3 nivåer med begrunnelse.
   - Ex: sikkerhet: "Grunnleggende (RLS + HTTPS)", "Middels (+ 2FA + audit-log)", "Enterprise (+ SOC2 + pen-test)"
```

**AskUserQuestion**: ett kall med 4 spørsmål (ett per dimensjon — ytelse, skala, sikkerhet, i18n) hvor hver bruker agentens 3 nivåer.

**Fil-output** (`04-ikke-funksjonelle-krav.md`).

---

### Fase 05 — User flows

**Research-agent prompt per Must-have-feature**:
```
Feature: "<navn>"

1. Søk etter typiske brukerflyt-patterns for denne type handling.
2. Foreslå 3 ulike flyt-varianter:
   - "Direct" (fewest steps)
   - "Guided" (wizard med validering per steg)
   - "Smart" (AI/automatikk-assistert)
3. For hver: steg-for-steg, edge cases, UX-avveininger.
```

**AskUserQuestion per feature**: velg flyt-variant.

**Fil-output** (`05-user-flows.md`): happy path + edge cases per feature.

---

### Fase 06 — Datamodell

**Research-agent prompt**:
```
Gitt features fra 03-funksjonelle-krav.md og flows fra 05-user-flows.md:

1. Identifisér kandidat-entiteter (tabeller).
2. Foreslå 2-3 ulike datamodell-strukturer:
   - "Normalisert" (3NF, minimal duplisering)
   - "Denormalisert" (optimalisert for lesing, mer redundans)
   - "Hybrid" (normalisert + read-replicas/views)
3. For hver: ER-diagram-skisse (tekstuelt), RLS-strategi, indekserings-strategi, trade-offs.

Spesielt relevant for Supabase: husk RLS på alle brukerdata-tabeller.
```

**AskUserQuestion**: velg datamodell-strategi.

**Oppfølging per entitet**:
- Agent foreslår felter + typer + constraints
- Bruker godtar eller endrer

**Fil-output** (`06-datamodell.md`): entities, relasjoner, RLS-strategi, indekser.

---

### Fase 07 — Arkitektur

**Research-agent prompt**:
```
Gitt alt tidligere: visjon, persona, scope, features, datamodell.

1. Hvilke ekstra integrasjoner er aktuelle? (Stripe, Resend, Sentry, analytics, osv.)
2. Background jobs — trenger vi? (cron, queues, webhooks)
3. Foreslå 3 arkitektur-tilnærminger:
   - "Minimal" (bare Supabase + Vercel + Resend for e-post)
   - "Standard" (+ Stripe + Sentry + posthog)
   - "Advanced" (+ egne edge-functions, real-time, Inngest for jobs)

Begrunn hver mot prosjektets ikke-funksjonelle krav (fase 04).
```

**AskUserQuestion**: velg arkitektur-nivå.

**Oppfølging**: per valgte integrasjon — spør om spesifikk tjenesteleverandør hvis relevant (Stripe vs Vipps, Resend vs Postmark).

**Fil-output** (`07-arkitektur.md`): stack + integrasjoner + første ADR-er.

---

### Fase 08 — Roadmap

**Research-agent prompt**:
```
Gitt alt tidligere:
- Scope (Must/Should/Could)
- Arkitektur-valg
- Brukers tidshorisont (spør først)

1. Foreslå 3 roadmap-strategier:
   - "Sprint" (MVP på 1-2 uker, aggressiv)
   - "Iterativ" (MVP på 4-8 uker, bygge inn læring per iterasjon)
   - "Konservativ" (MVP på 2-3 mnd, mer polering før launch)
2. For hver: fase-inndeling, milepæler, exit-kriterier, risiko.
```

**AskUserQuestion**: velg strategi + tidshorisont.

**Fil-output** (`08-roadmap.md`): faser med exit-kriterier.

---

### Fase 09 — Åpne spørsmål

**Agent-oppgave** (ikke AskUserQuestion her — agenten samler):
```
Les alle 9 tidligere filer. Identifisér:
1. Tekniske risikoer (hva kan gå galt, sannsynlighet, konsekvens, mitigering)
2. Antagelser som må valideres (marketsfit, teknologi-valg, skaleringsantagelser)
3. TBD-felter fra tidligere faser
4. Stakeholders som må involveres (juridisk, design, forretning)

Produsér `09-aapne-sporsmal.md` direkte — ingen AskUserQuestion nødvendig her.
Bruker kan senere lese og legge til.
```

## Avslutning

Etter alle 10 faser:

1. **Rapport til bruker**:
   ```
   ✓ Discovery ferdig — 10 hovedfiler + 9 research-filer i teknisk/dokumentasjon/prosjekt/
   
   Oppsummering:
   - Visjon: <valgt vinkling>
   - Primær-persona: <valg>
   - Scope: <X Must-have, Y Should-have features>
   - Tidsramme: <valgt>
   - Arkitektur-nivå: <valg>
   - <N> åpne spørsmål / antagelser å validere
   
   Anbefalt neste steg:
   1. Les gjennom filene, juster det som ikke føles riktig
   2. Valider antagelser fra 09-aapne-sporsmal.md før betydelig kode skrives
   3. Start første Must-have via /2.0-task <feature-navn>
   4. Re-kjør /7.0-discovery hvis scope eller visjon endrer seg
   ```

2. **Commit** (valgfritt — spør bruker):
   - Hvis ja: stage filer, commit med melding "docs(prosjekt): initial discovery (10 filer)"

3. **Oppdater CLAUDE.md** (spør bruker):
   - Hvis ja: legg til kort prosjekt-oppsummering øverst (etter `{{PROJECT_DESCRIPTION}}`) som Claude alltid vil se

## Feilsøking

- **Agent gir ikke gode anbefalinger**: prompt-en er kanskje for åpen. Legg til mer spesifikk kontekst (f.eks. konkurrenter brukeren kjenner til) og re-spawn.
- **WebSearch feiler**: fallback til WebFetch mot kjente docs-URLer (competitors, industry blogs).
- **Bruker velger alltid "Other"**: agenten har ikke truffet hjemme. Still et avklarende AskUserQuestion om hva som er viktig før neste forsøk.

## Re-kjøring

Ved re-kjøring spør Claude først:
- **Hva endrer seg?** (alle 10 faser / bare én / bare noen)
- Kjør kun de valgte fasene på nytt, behold andre filer intakt.

## Anti-patterns

- ❌ Hopp over research-agent — gir generiske alternativer
- ❌ Gi mer enn 4 alternativer per AskUserQuestion — verktøyet støtter maks 4 (pluss Other)
- ❌ Spør bruker om ting som allerede er besvart (duplisering er frustrerende)
- ❌ Gå videre før bruker bekrefter valg — la dem reflektere
- ❌ Hardkode lenker / kilder uten å fetche — bruk WebSearch + WebFetch
