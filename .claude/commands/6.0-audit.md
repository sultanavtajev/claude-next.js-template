## Full audit

**Scope**: $ARGUMENTS (hvis tomt: hele `src/`)

Denne kommandoen kjører alle fire audit-agents i sekvens, samler funnene, og skriver én samlet rapport. Brukes før større deploys, før PR-merge, eller periodisk.

## Workflow

### 1. Bestem scope

Hvis `$ARGUMENTS` er tomt: audit hele `src/`.
Hvis spesifikk path gitt: audit kun det området (pluss globale ting som dependencies og env).

### 2. Kjør agents i rekkefølge

Bruk `Agent`-verktøyet, ett kall per audit:

```
Agent({subagent_type: "nextjs-security-auditor", prompt: "Audit <scope>. Se nextjs-security-auditor.md for sjekkliste."})
Agent({subagent_type: "nextjs-a11y-auditor", prompt: "Audit <scope>. Se nextjs-a11y-auditor.md for sjekkliste."})
Agent({subagent_type: "nextjs-performance-auditor", prompt: "Audit <scope>. Se nextjs-performance-auditor.md for sjekkliste."})
Agent({subagent_type: "nextjs-reviewer", prompt: "Review <scope>. Fokus på design-system + kode-kvalitet."})
```

Kan kjøres parallelt (én message med flere Agent-calls) for effektivitet.

### 3. Samle funnene

Konsolider output i `teknisk/dokumentasjon/audits/YYYY-MM-DD-full.md`:

```markdown
# Full Audit — <dato>

Scope: <scope>
Utført av: Claude Code (4 agents)

## Sammendrag

- 🚨 Kritisk: <antall>
- ⚠️ Forbedringer: <antall>
- ✓ OK: <antall>

## Security ([full rapport fra nextjs-security-auditor])

### 🚨 Kritisk
...

### ⚠️ Forbedring
...

## A11y (WCAG 2.2 AA)
...

## Performance
...

## Kode-kvalitet (design-system, Next.js-mønstre)
...

## Anbefalt handling

1. Fiks kritiske funn før deploy
2. Lag issues for forbedringer
3. Runtime-sjekker å kjøre: Lighthouse, axe DevTools, `pnpm audit`
```

### 4. Rapporter til bruker

Kort oppsummering med:
- Lenke til full rapport i `teknisk/dokumentasjon/audits/`
- Antall kritiske / forbedringer / OK per kategori
- Anbefalt handling

Hvis kritiske funn: foreslå å fikse før deploy.

## Valgfrie parametere

Bruker kan spesifisere kun én kategori ved å bruke sub-kommandoene:
- `/6.1-a11y` — bare a11y
- `/6.2-security` — bare security
- `/6.3-performance` — bare performance

## Etter audit

Bruker velger:
1. **Fikse nå**: start `/2.0-task fix audit findings` med kritiske funn som sjekkliste
2. **Logge som issues**: opprett GitHub-issues via `gh issue create`
3. **Ignorere**: dokumentér begrunnelse i `teknisk/dokumentasjon/beslutninger/` (eller senere i ADR-log)
