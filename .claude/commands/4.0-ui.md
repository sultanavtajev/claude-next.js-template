## Krav til Claude (les før du starter)

Denne kommandoen finnes spesifikt for å garantere at to ting skjer **før** du genererer kode:

1. **`design-system-retrieval`-skillen aktiveres** — du skal lese `design-system/MASTER.md` (+ eventuell side-override) og rapportere hva som er lest.
2. **`ui-ux-pro-max`-skillen kalles** — ikke generér UI uten å bruke dens design-intelligens (palletter, font-pairings, layout-mønstre, komponent-anbefalinger). Selv om oppgaven virker triviell.

## Workflow

### 1. Les designsystemet

```bash
cat design-system/MASTER.md
```

Hvis oppgaven gjelder en spesifikk side, sjekk også:
```bash
ls design-system/pages/
```

Hvis `design-system/pages/<slug>.md` finnes for siden du skal endre, les den — den overstyrer MASTER på de spesifikke punktene.

Hvis MASTER ikke finnes: stopp og rapportér at oppstart-steg 04 må kjøres først.

### 2. Konsulter ui-ux-pro-max

Bruk skillen til å hente design-anbefalinger som ligger innenfor MASTER-rammene. Eksempler på når du trenger den:
- Hvilke shadcn-komponenter passer best til den layouten?
- Hvilken font-vekt + størrelse for en hero-overskrift i minimalism-stil?
- Hvordan plassere call-to-action-knappen for maksimal konvertering uten å bryte minimalism?

`ui-ux-pro-max` har 161 palletter, 67 stiler, 57 font-pairings + UX-guidelines. Bruk det.

### 3. Spør bruker om side-override (hvis nødvendig)

Hvis oppgaven *krever* avvik fra MASTER (f.eks. "lag en bold kampanje-landing som er anner­ledes enn resten av appen"):

- Spør via `AskUserQuestion`: skal dette låses som permanent override?
- Hvis ja: kjør `python3 .claude/skills/ui-ux-pro-max/scripts/search.py "<query>" --design-system --persist --page "<slug>"` for å opprette `design-system/pages/<slug>.md`.
- Hvis nei: gjør en éngangsmodifikasjon men ikke lås den — dokumentér i kommentar at det er bevisst avvik.

### 4. Rapportér før kodegenerering

```
Designsystem innlest:
- MASTER: design-system/MASTER.md ✓
- Side-override: <fil eller "ingen">
- ui-ux-pro-max konsultert for: <hva spesifikt>

Planlagt generering:
- <kort beskrivelse av komponent/side/endring>
- Komponenter brukt: <liste fra src/components/ui/ + nye>
- Farger: via CSS-variabler fra globals.css
- Fonts: <fra MASTER>
```

### 5. Generér koden

- Bruk shadcn-komponenter (eller utvid eksisterende) der det passer
- Aldri hardkod farger — bruk CSS-variabler (`bg-primary`, `text-foreground`, osv.)
- All brukervendt tekst gjennom `useTranslations()` / `getTranslations()` (se i18n-translations-skill)
- Mobile-first responsivitet
- A11y: riktige role/aria, label tilkoblet input, alt-tekst på bilder

### 6. Verifiser

- `pnpm typecheck` + `pnpm lint`
- Test i nettleser via `pnpm dev` (med `run_in_background: true` + curl, ikke sleep)
- Eventuelt kjør `nextjs-reviewer`-agent for å sjekke design-system-etterlevelse

## Hvorfor denne kommandoen finnes

`ui-ux-pro-max` aktiveres ofte automatisk for store oppgaver ("bygg dashboard"), men kan hoppes over for små ("endre den knappen til blå"). Resultat: Claude lager improviserte design som ikke matcher låst stil.

`/4.0-ui` garanterer at *både* MASTER-konsultering *og* ui-ux-pro-max kalles, uansett hvor liten oppgaven er. Bruk den når du er usikker på om Claude vil følge designsystemet automatisk.

## Når du IKKE trenger denne kommandoen

- For oppgaver som ikke involverer UI (database, API, server logikk).
- For trivielle ting som rename av prop eller variabel uten visuell effekt.
- Inne i `/2.0-task`/`/2.1-task-team` — de har allerede UI-instruks som peker hit.
