## Security-audit

**Scope**: $ARGUMENTS (default: hele `src/` + `supabase/` + env)

Kjør `nextjs-security-auditor`-agenten mot gitt scope. Brukes før deploy til prod, periodisk (månedlig), eller når ny auth/admin-funksjonalitet er lagt til.

### Workflow

1. Bekreft scope.
2. Les `teknisk/dokumentasjon/supabase-snapshot.md` først (for RLS-oversikt) — regenerer med `pnpm db:snapshot` hvis utdatert.
3. Kall `Agent({subagent_type: "nextjs-security-auditor", prompt: "Audit <scope>. Følg sjekklisten i nextjs-security-auditor.md."})`.
4. Lagre rapport til `teknisk/dokumentasjon/audits/YYYY-MM-DD-security.md`.
5. Kjør også `pnpm audit --prod` for dependency-vulnerabilities og inkluder output i rapporten.

### Kritisk vs forbedring

**Kritisk (må fikses før deploy)**:
- RLS mangler på tabeller med brukerdata
- Service role-key eksponert i klient-kode
- Hardkodede secrets
- `getSession()` brukt server-side
- Webhook uten signature-verifisering

**Forbedring (bør fikses snart)**:
- Rate-limit mangler
- Ufullstendig input-validering
- Svake passord-policies
- Manglende `Content-Security-Policy`-headers

### Etter audit

- Ved kritiske funn: **ikke deploy** før fiks. Tilby `/2.0-task fix security findings`.
- Ved dep-vulnerabilities: `pnpm audit --fix` og regenerér lockfile.
- Dokumentér bevisste unntak i `teknisk/dokumentasjon/beslutninger/<nr>-<tema>.md`.
