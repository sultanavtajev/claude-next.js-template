## 🤖 Claude (15:21)

Kort forklart:

- **Skills**: Gjenbrukbare "ferdigheter" med instruksjoner (+ valgfrie scripts/ressurser) som Claude laster inn på eget initiativ når oppgaven matcher. Modell-drevet.
- **Commands** (slash commands, f.eks. `/commit`): Brukerutløste prompt-maler. Du skriver `/navn` og det ekspanderes til en ferdig prompt. Bruker-drevet.
- **Agents** (subagents, f.eks. `Explore`, `Plan`): Egne Claude-instanser med eget verktøysett, kontekstvindu og systemprompt. Kalles via `Agent`-verktøyet for å parallellisere eller skjerme hovedkonteksten.
- **Rules** (CLAUDE.md / memory): Statisk tekst som alltid lastes inn i konteksten — prosjekt- eller brukerregler som styrer oppførsel uten å kjøre kode.
- **Hooks**: Shell-kommandoer som *harnesset* (ikke Claude) kjører automatisk på events som `PreToolUse`, `PostToolUse`, `UserPromptSubmit` osv. Deterministisk automatisering — kjører uansett hva modellen "vil".

Tommelfingerregel: **Rules** = alltid på, **Skills** = modellen velger, **Commands** = du velger, **Agents** = delegert delarbeid, **Hooks** = garantert kjøring rundt verktøykall.

## Anbefalt oppsett

### 1. `CLAUDE.md` i prosjektroten (Rules)
Alltid-på kontekst. Hold den **kort** (lastes hver gang):
- Tech stack (Next.js 15, App Router, TypeScript, Tailwind, osv.)
- Mappestruktur og konvensjoner (`app/`, `components/`, `lib/`)
- Kommandoer (`pnpm dev`, `pnpm test`, `pnpm lint`)
- Harde regler ("bruk Server Components som default", "ingen `any`", "Zod for all input-validering")

### 2. Hooks i `.claude/settings.json` (deterministisk)
For ting som *må* skje:
- `PostToolUse` på Edit/Write → kjør `prettier` + `eslint --fix` på endrede filer
- `PreToolUse` → blokker redigering av `.env`, `node_modules`, `.next`
- `UserPromptSubmit` → injiser f.eks. gjeldende git-branch

### 3. Skills i `.claude/skills/` (modell-valgt)
For spesialisert kunnskap som *bare* trengs noen ganger:
- `nextjs-server-action` — mal + regler for Server Actions
- `nextjs-route-handler` — API route-konvensjoner
- `shadcn-component` — hvordan legge til/style komponenter
- `prisma-migration` — migration-flyt

Skills er bedre enn å stappe alt i CLAUDE.md fordi de bare lastes ved behov.

### 4. Slash commands i `.claude/commands/` (bruker-utløst)
Snarveier for repeterende workflows:
- `/new-route <navn>` — scaffolder route + test
- `/new-component <navn>` — komponent + story + test
- `/review` — kjør lint + typecheck + test og oppsummer

### 5. Subagents i `.claude/agents/` (valgfritt)
Kun hvis du har gjentakende delegerbare oppgaver, f.eks.:
- `nextjs-reviewer` — sjekker Server/Client component-bruk, `use client`-grenser, caching
- `accessibility-checker` — granske JSX for a11y

## Distribusjon som template
Legg hele `.claude/`-mappen + `CLAUDE.md` i et template-repo. Da får hvert nytt prosjekt samme oppsett ved `git clone` eller `create-next-app --example`.

## Hva repoet er
Et stort, modent oppsett (47 agents, 181 skills, hooks, språk-spesifikke rules, plugin-manifest) som har samlet alt Claude Code kan gjøre i én distribuerbar pakke. Hovedbudskapet er at **skills** nå er den primære overflaten — slash commands er degradert til "legacy shims" som bare peker videre til skills.

## Interessante grep du kan stjele

1. **`rules/` per språk** — i stedet for én diger CLAUDE.md, splitt regler i `rules/common/`, `rules/typescript/`, `rules/nextjs/` osv. og last inn det relevante. Holder kontekstvinduet slankt.

2. **Plugin-manifest (`.claude-plugin/`)** — Claude Code støtter plugins/marketplace. Hvis du vil dele templaten på tvers av maskiner eller team, pakk `.claude/`-mappen som et plugin i stedet for å kopiere filer manuelt.

3. **Install-script (`install.sh --profile full`)** — de har profiler (minimal/full) fordi rules ikke kan distribueres via plugin-systemet. Verdt å kopiere mønsteret: ett script som setter opp `.claude/` + CLAUDE.md i et nytt prosjekt.

4. **Hooks for memory/læring** — de bruker `SessionStart`/`SessionEnd`-hooks for å persistere kontekst og auto-ekstrahere mønstre til nye skills. Mer ambisiøst enn du trenger, men idéen "hook som skriver til memory" er nyttig.

5. **Skills > Commands** — bekrefter rådet jeg ga: bygg Next.js-spesifikk kunnskap som skills (`nextjs-server-action`, `nextjs-caching`, `shadcn-add`), ikke som slash commands. Modellen henter dem ved behov.

6. **Spesialiserte reviewer-agents** — `code-reviewer`, `security-reviewer`, språk-reviewere. For Next.js kunne du hatt en `nextjs-reviewer` som spesifikt sjekker Server/Client-grenser, `use client`, caching-direktiver og `next/image`-bruk.

- Multi-model runtime (`ccg-workflow`) — egen avhengighet, ikke nødvendig.

1. **`examples/saas-nextjs-CLAUDE.md`** finnes — verdt å lese som referanse for hvordan en ekte Next.js+Supabase+Stripe CLAUDE.md ser ut. Sannsynligvis et godt utgangspunkt å klippe fra.

5. **Skills splittes i par**: `*-patterns/` + `*-testing/` + `*-security/` + `*-verification/`. Konsistent firkløver per stack. For Next.js kunne du hatt `nextjs-patterns`, `nextjs-testing`, `nextjs-security`, `nextjs-verification`.

6. **`agents/build-error-resolver.md` + språk-spesifikke build-resolvers** — egen agent kun for å løse byggefeil. Next.js har nok særegne feil (`use client`-grenser, hydration, Turbopack) til å rettferdiggjøre en `nextjs-build-resolver`.

  - **`CLAUDE.md`** (rot) = instruksjoner/kunnskap for Claude. Én fil.

| Plassering | Rolle |

## 👤 Bruker (10:46)

Er det slik at skillen automatisk dokumenterer stilen for alle sider under design-system/pages/

## 🤖 Claude (10:46)

Nei. `design-system/pages/` er **ikke** auto-populert. Mappen er tom ved bootstrap og fylles kun når du eksplisitt ber om en override.

## Hvordan det faktisk fungerer

| Fil | Når opprettes den | Inneholder |
|-----|-------------------|------------|
| `design-system/MASTER.md` | Én gang i steg 04 | **Alt** — global baseline for hele appen (farger, typografi, spacing, komponenter, tilstander) |
| `design-system/pages/<slug>.md` | Kun når du ber om det | **Bare avvik** fra MASTER for den ene siden |

## Hvorfor ikke auto-generere alle sider

Tanken bak mønsteret er:
- **MASTER er standarden.** 95 % av sider skal følge den uten avvik.
- **Overrides er unntaket.** De lages kun for sider som *bevisst* skal se annerledes ut (f.eks. en bold kampanje-landing i en ellers minimalistisk app, eller en kompakt print-versjon av en rapport).
- **Ingen støy.** Hvis hver side fikk sin egen fil automatisk, ville Claude måtte lese N filer før hver generering uten at de tilfører noe — 99 % av innholdet ville bare være "bruk MASTER".

## Når lages en override-fil

Tre scenarier:

1. **Bruker ber eksplisitt**: "Lag en override for checkout-siden — mer kompakt layout" → Claude kjører `search.py ... --page "checkout"`.
2. **Claude oppdager et reelt avvik under UI-generering**: f.eks. bruker ber om en side som ikke kan bygges uten å bryte MASTER (eks: "lag en dashboard-landing med neumorphism"). Da foreslår `design-system-retrieval`-skillen å opprette en override i stedet for å improvisere.
3. **Retroaktiv formalisering**: hvis en side har drift fra MASTER som er blitt akseptert, kan du låse avviket i en override-fil slik at det er dokumentert, ikke tilfeldig.

## Effekten

Ved start har du `design-system/pages/` tom eller med én placeholder-fil. Over tid vokser den organisk med ekte avvik — ikke auto-støy. Hver fil er et bevisst designvalg.

Hvis du ønsket en mer preskriptiv modell (f.eks. "Claude skal alltid opprette en override-fil for hver ny rute brukeren lager"), kan vi kodifisere det i `design-system-retrieval`-skillen. Men standardmønsteret fra skillens README er bevisst pull-based.

Vi bruker **pnpm** i templaten gjennomgående — ikke npm. Her er forskjellene og hvorfor valget ble pnpm.

## npm vs pnpm — kort versjon

| | npm | pnpm |
|---|-----|------|
| **Installasjonsmetode** | Kopierer pakker til hver `node_modules/` | Lenker til ett globalt lager (`~/.pnpm-store/`) |
| **Diskplass** | ~500 MB per prosjekt (duplikatene ligger alle steder) | ~50 MB per prosjekt (hardlinks) |
| **Installasjonstid** | Saktere (kopiering) | 2–3× raskere (hardlinks) |
| **Strict mode** | Flate `node_modules/` — pakker kan "lekke" mellom dependencies (phantom deps) | Strikt — bare deklarerte dependencies er tilgjengelige |
| **Monorepo** | Krever `npm workspaces` | Innebygd workspace-støtte, raskere |
| **Lockfile** | `package-lock.json` | `pnpm-lock.yaml` |
| **Preinstallert** | Kommer med Node.js | Må installeres: `npm i -g pnpm` |

## Hvorfor templaten bruker pnpm

1. **Fart**: installerer Supabase + shadcn + alle dep-ene på sekunder i stedet for minutter.
2. **Diskplass**: du sa du har 25+ Next.js-prosjekter i `C:\Kodeprosjekter\`. Med npm ville hvert prosjekt ha sin egen kopi av `react`, `next`, `tailwind` osv. (~500 MB × 25 = 12+ GB). pnpm deler dette via hardlinks.
3. **Strict dependencies**: fanger opp bugs der du ved et uhell importerer en pakke som er transitiv (ikke direkte deklarert). Disse bryter når dep-treet endres senere.
4. **Next.js community standard**: nextjs.org/docs bruker `pnpm` i eksemplene, `create-next-app --use-pnpm` er førsteklasses støttet.

## Forutsetning

For at templaten skal virke må brukeren ha pnpm installert én gang på maskinen:

```bash
npm i -g pnpm
```

Dette ligger dokumentert i README under "Forutsetninger på maskinen" (tidligere — nå i TEMPLATE.md).

## Kan vi bytte til npm hvis ønsket?

Ja. Det ville kreve:
- Fjerne `--use-pnpm` fra `create-next-app`-flaggene i oppstart-steg 02
- Erstatte `pnpm`-kommandoer med `npm run` i alle stegfiler, slash-kommandoer, og CLAUDE.md
- Bytte `pnpm add` → `npm install` i Supabase-steget og andre install-kommandoer
- Bytte `pnpm-lock.yaml` → `package-lock.json`

## Biblioteket

For Next.js App Router er **`next-intl`** de-facto standard i 2025/2026:
- Native støtte for Server Components (oversettelser kjører serverside)
- Dynamisk route segment `/[locale]/...`
- JSON-message-filer per locale
- God TypeScript-støtte
- Aktivt vedlikeholdt

## Strukturell utfordring

next-intl restrukturerer `src/app/...` → `src/app/[locale]/...`, og introduserer sin egen `proxy.ts` (tidligere middleware) for locale-deteksjon og redirects. Dette **konflikterer med** Supabase sin `proxy.ts` for session-refresh.

Løsningen: én felles `src/proxy.ts` som chainer begge:
```ts
import { createI18nProxy } from 'next-intl/proxy';
import { updateSession } from '@/lib/supabase/proxy';

const handleI18n = createI18nProxy(routing);

export async function proxy(request) {
  const response = await handleI18n(request);
  return updateSession(request, response);  // chainer session-refresh etter locale-routing
}
```

Derfor må **i18n settes opp før Supabase**, ikke etter.

La meg forklare hvordan hook-systemet fungerer i templaten.

## Arkitekturen

Claude Code har et hook-system der `.json`-konfig i `.claude/settings.json` kobler **events** til **scripts**. Scriptene kan være hva som helst som kan kjøres fra shell — hos oss Node.js-scripts som tar input fra `stdin`.

## To event-typer templaten bruker

### `PreToolUse` — kjører FØR Claude eksekverer et verktøy

Kan **blokkere** verktøyet hvis scriptet avslutter med exit code `2`. Brukes til sikkerhet og validering.

**Vår bruk**: `pre-edit-guard.js` blokkerer Edit/Write på `.env`, `.next/`, `node_modules/`, `.git/`.

### `PostToolUse` — kjører ETTER verktøyet er ferdig

Kan **ikke** blokkere (verktøyet er allerede kjørt). Brukes til formatering, sidegadgets, regenerering.

**Vår bruk**:
- `post-edit-format.js` — kjører prettier + eslint på filer Claude har redigert
- `post-migration-snapshot.js` — regenererer Supabase-snapshot når migrasjoner endres

## Konfigurasjonen i `settings.json`

```json
"hooks": {
  "PreToolUse": [
    {
      "matcher": "Edit|Write",        ← matcher regex mot tool-navn
      "hooks": [
        {
          "type": "command",
          "command": "node $CLAUDE_PROJECT_DIR/.claude/hooks/scripts/pre-edit-guard.js"
        }
      ]
    }
  ],
  "PostToolUse": [
    {
      "matcher": "Edit|Write",
      "hooks": [
        { "type": "command", "command": "node $CLAUDE_PROJECT_DIR/.claude/hooks/scripts/post-edit-format.js" },
        { "type": "command", "command": "node $CLAUDE_PROJECT_DIR/.claude/hooks/scripts/post-migration-snapshot.js" }
      ]
    }
  ]
}
```

- **`matcher`**: regex mot tool-navnet (`Edit`, `Write`, `Bash`, osv.). `"Edit|Write"` = kjør for begge.
- **`hooks`-array**: flere scripts kan kjøres på samme event. Templaten har to post-edit-scripts som kjører sekvensielt.
- **`command`**: shell-kommando Claude Code kjører. Bruker `$CLAUDE_PROJECT_DIR` (env-var Claude Code setter til prosjektrotmappen).

## Hvordan scriptene kommuniserer med Claude

### Input via stdin (JSON)

Claude Code sender tool-konteksten som JSON på stdin:

```json
{
  "tool_name": "Edit",
  "tool_input": {
    "file_path": "C:/Kodeprosjekter/mitt-prosjekt/src/app/page.tsx",
    "old_string": "...",
    "new_string": "..."
  },
  "tool_response": { "success": true }
}
```

Scriptet leser det:
```javascript
let input = "";
process.stdin.on("data", (chunk) => (input += chunk));
process.stdin.on("end", () => {
  const payload = JSON.parse(input);
  const filePath = payload?.tool_input?.file_path;
```

### Output til Claude via stderr

Hvis scriptet skriver til `stderr`, ser Claude det. Brukes for å fortelle Claude hvorfor noe ble blokkert (PreToolUse) eller hva som skjedde (PostToolUse).

### Exit codes

- `exit(0)` — OK, fortsett
- `exit(2)` — **blokker** (bare gyldig for PreToolUse). Claude ser stderr-meldingen og vet den ikke får lov til å utføre verktøyet.

## Konkret eksempel: vår `post-migration-snapshot.js`

```javascript
// 1. Les stdin
let input = "";
process.stdin.on("data", (chunk) => (input += chunk));
process.stdin.on("end", () => {
  try {
    // 2. Parse JSON
    const payload = JSON.parse(input || "{}");
    const filePath = payload?.tool_input?.file_path;
    if (!filePath) process.exit(0);

    // 3. Sjekk path-mønster
    const projectRoot = process.env.CLAUDE_PROJECT_DIR || process.cwd();
    const rel = path.relative(projectRoot, filePath).replace(/\\/g, "/");
    if (!rel.startsWith("supabase/migrations/") || !rel.endsWith(".sql")) {
      process.exit(0);   // ikke en migrasjon — exit stille
    }

    // 4. Kjør snapshot i bakgrunn (blokkerer ikke Claude)
    const child = spawn("pnpm", ["db:snapshot"], {
      cwd: projectRoot,
      detached: true,
      stdio: "ignore",
      shell: true,
    });
    child.unref();

    process.stderr.write(`[post-migration-snapshot] regenererer...\n`);
  } catch {}
  process.exit(0);   // alltid 0 — snapshot-regenerering skal aldri blokkere
});
```

## Viktig å vite

- **Hooks kjører i shell** — de er vanlige prosesser, så de kan gjøre hva som helst (database-kall, nettverk, filoperasjoner).
- **PostToolUse kan være sakte** — Claude venter på at alle hooks fullfører før den fortsetter. Derfor bruker vi `spawn + unref` for snapshot-regenerering slik at det kjører detached.
- **Feil i hooks er "silent" by default** — exit code + stderr er eneste måte å kommunisere. Claude rapporterer ikke automatisk hook-feil til bruker.
- **Matchers er per-tool-navn** — ikke per filsti. Scriptet må selv sjekke `tool_input.file_path` hvis det bare skal reagere på visse filer (slik `post-migration-snapshot.js` gjør).

## Implementasjon

Stop-hook gjør dette:
1. Finner `transcript_path` i hook-payload

## Fullt hook-oversikt (seks scripts, fem event-kategorier)

| Event | Script | Hva det gjør |
|-------|--------|--------------|
| **SessionStart** | `session-start.js` | Sender briefing (git-state, commits, snapshot-alder, aktive sjekklister, env-vars) til Claude via `additionalContext` |
| **PreToolUse** (Edit/Write) | `pre-edit-guard.js` | Blokkerer redigering av `.env`, `.next/`, `node_modules/`, `.git/` |
| **PostToolUse** (Edit/Write) | `post-edit-format.js` | Kjører prettier + eslint --fix på endret fil |
| **PostToolUse** (Edit/Write) | `post-migration-snapshot.js` | Kjører `pnpm db:snapshot` når `supabase/migrations/*.sql` endres |
| **Stop** | `stop-chat-archive.js` | Lagrer full chat-historikk til `teknisk/dokumentasjon/chats/raw/` + månedsindeks |
| **PreCompact** | `pre-compact.js` | Dumper sesjon-state til `teknisk/dokumentasjon/session-state.md` før kompaktering |

## Kontinuitets-flyten

```
Start claude
  └─> SessionStart-hook → briefing med state
      └─> Claude jobber, redigerer filer
          ├─> Pre-edit-guard sikrer mot sensitive filer
          └─> Post-edit-format + snapshot holder ting ryddig
      └─> Kontekst nærmer seg grensen
          └─> PreCompact-hook → dumper state til fil
      └─> Claude svarer ferdig
          └─> Stop-hook → lagrer full chat-historikk
  
Neste gang claude starter (samme eller neste dag):
  └─> SessionStart-hook → viser at session-state.md finnes
      └─> Claude leser session-state.md → vet hvor vi var
      └─> Claude kan søke i teknisk/dokumentasjon/chats/ for historisk kontekst
```

Hele verdikjeden henger sammen: hookene gir Claude persistent minne på tvers av sesjoner via filsystemet.

## Top 6 anbefalinger

### 1. Flere spesialiserte subagents (høy verdi, lav innsats)
I tillegg til `nextjs-reviewer` og `nextjs-build-resolver`:
- **`security-auditor`** — OWASP-sjekk, Supabase-RLS-gaps, env-lekkasjer, secret-detection
- **`a11y-auditor`** — WCAG 2.2, keyboard-nav, screen-reader-struktur, color-contrast
- **`performance-auditor`** — bundle-størrelse, Core Web Vitals, unødvendig client-JS, caching
- **`migration-planner`** — når bruker vil endre DB-schema: agent planlegger migrations-rekkefølge med rollback-strategi *før* noe skrives

Hver kan kalles isolert via `Agent({subagent_type: "..."})`, eller via en master-kommando.

### 2. `/6.0-audit` — kjører alle reviewers i sekvens
Ett kall som kjører `nextjs-reviewer` + `security-auditor` + `a11y-auditor` + `performance-auditor` og produserer én samlet rapport. Brukes før større commits / PR. Supplert med `/6.1-a11y`, `/6.2-security`, osv. for enkelt-audits.

### 3. Next.js-spesifikke skills vi mangler
- **`form-patterns`** — react-hook-form + Zod + Server Actions + useActionState (30 % av Next.js-arbeid er forms)
- **`caching-strategy`** — Next.js 15+ cache-semantikk er kompleks; skill som forklarer `revalidateTag`/`revalidatePath`/`cache: "no-store"`/`dynamic = "force-dynamic"`-kombinasjoner
- **`error-boundaries`** — `error.tsx`, `not-found.tsx`, `global-error.tsx`-mønstre med Sentry-integrasjon hvis brukt
- **`streaming-suspense`** — når bruker Suspense-boundaries med Server Components, `loading.tsx`-mønstre

### 4. Arkitektur-beslutninger (ADR-pattern)
Ny konvensjon i `teknisk/dokumentasjon/beslutninger/`:
